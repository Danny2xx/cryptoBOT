Part 3 — This Week (W3): Your To-Do List

3.1 Action 1 — Close out the TimescaleDB / Redis decision (do this first, today/tomorrow)

This is action item #5 from the kickoff meeting: "Decide TimescaleDB and Redis infrastructure before Epic 1." It sounds bigger than it is — the BRD has already made most of this decision; you mainly need to confirm the details and write them down so the action item is closed.

What these things actually are (plain English)

  • TimescaleDB = PostgreSQL (a normal relational database) plus an extension that's optimised for time-series data. You'll use it to store market prices/candles, order book snapshots, and the state of orders/positions/config. Think of it as the bot's memory.
  • Redis = an in-memory data store used here as an event bus — a fast messaging layer so that, say, the market-data component can broadcast "new price tick" and the strategy engine reacts instantly, without the two components needing to know about each other directly. Think of it as the bot's nervous system.

The decision (recommended)

  1. Event bus: Redis, not NATS. The BRD lists "Redis / NATS" as options — Redis is simpler to operate, has a mature Python client (redis-py), and easily handles the >1000 events/sec performance target at this scale. NATS adds operational complexity with no benefit here.
  2. Both run as Docker containers, defined in docker-compose.yml alongside your app skeleton — this is also literally what Epic 1's acceptance criteria (E1-US2) requires:
    • timescale/timescaledb:latest-pg16 — official TimescaleDB image (bundles Postgres 16)
    • redis:7-alpine — official, lightweight Redis image
  3. Where it runs for now: locally, on your machine, via Docker Desktop — fine for development, backtesting, and paper trading.
Open question to raise this week: BRD §10 assumes "a self-hosted Linux VPS or homelab is available for always-on components." From Epic 2 onward, market-data ingestion needs to run continuously (24/7) — your laptop can't do that. If nobody has actually provisioned a VPS/homelab yet, flag this now in the Team 1 - CTB channel. Setting one up has lead time, and it'll block you in ~2 weeks if it's not sorted.

How to close the action item

Part 4 — Docker & Infra Crash Course (you said this is new — start here)

4.1 The four words that matter

  • Image — a frozen snapshot of an app + everything it needs to run (like a ZIP file that contains a tiny computer). E.g. "redis:7-alpine" is a ready-made image someone else built for you.
  • Container — a running instance of an image. You can start, stop, and delete containers without affecting the image they came from.
  • Dockerfile — a recipe for building your own image (used for your app skeleton, since nobody published "Daniel's trading bot" as an image).
  • docker-compose.yml — a file that describes a group of containers (your database, Redis, and app) and how they connect, so you can start everything with one command.

4.2 Why this project uses Docker

The BRD requires "Docker Compose on Linux VPS or self-hosted homelab" and "each application sandboxed in its own container." Practically, this means: TimescaleDB, Redis, and your Python app each run in their own isolated box, all wired together by docker-compose.yml. This is also exactly why your "decide infra" action item and Epic 1 overlap — deciding the infra is writing the Compose file.

4.3 Install Docker Desktop (Mac)

  1. Download Docker Desktop for Mac from Docker's official site (search "Docker Desktop download" — pick the build matching your Mac's chip, Apple Silicon or Intel).
  2. Install it like any other Mac app, then open it once — it needs to finish starting before docker commands work.
  3. Verify in Terminal: 
    docker --version
docker compose version
    Both should print version numbers.

4.4 Command cheat-sheet (the ~6 you'll actually use)

CommandWhat it does
docker compose up -dStart everything in the Compose file, in the background
docker compose downStop and remove all containers (data in volumes survives)
docker compose psList running containers and their status
docker compose logs -f appStream logs from the "app" service (Ctrl+C to stop watching)
docker compose exec app bashOpen a shell inside the running app container
docker compose buildRebuild your app image after changing the Dockerfile/code

engine/main.py (proves the skeleton actually works)

import os
import redis
import psycopg2


def main():
    r = redis.Redis(host=os.environ["REDIS_HOST"], port=int(os.environ["REDIS_PORT"]))
    r.ping()
    print("Redis OK")

    conn = psycopg2.connect(
        host=os.environ["POSTGRES_HOST"],
        port=os.environ["POSTGRES_PORT"],
        user=os.environ["POSTGRES_USER"],
        password=os.environ["POSTGRES_PASSWORD"],
        dbname=os.environ["POSTGRES_DB"],
    )
    print("Postgres OK")
    conn.close()


if __name__ == "__main__":
    main()

Part 6 — What's Coming Next (so you know the shape of the road)

Epic 2 — Market Data & Paper Trading (W4–W5)

Once Epic 0 picks a chain/DEX, you'll connect to it via RPC (web3.py or ethers) to read pool prices/reserves, normalise them into a shared MarketState format, and persist to TimescaleDB. Alongside that, you'll build a paper-trading simulator (realistic fills/fees/slippage, no real orders) and a feed-health check (alert if data goes stale >30s). This is gated by Epic 0 — don't start the RPC work until the venue is approved, but the paper simulator and MarketState schema can be designed in the meantime.

Epic 3 — Strategy Framework & Backtesting (W6–W7)

Define one common interface (receive(market_state) -> Optional[Orders]), then build three strategies behind it: rule-based (price-range orders), TA-signal (MA cross / RSI / Bollinger), and ML-driven. Build a backtester that runs the same strategy code against historical data, plus baseline strategies (buy-and-hold/no-op) for comparison (Sharpe, max drawdown, win rate).

Epic 4 — Risk Management & First Live Trades (W8)

Build the risk manager: hard capital cap, pre-trade checks (position size, exposure, drawdown, daily loss), a global kill switch (<1s activation), and state reconciliation on startup. Live trading is gated behind an explicit LIVE_ENABLED flag + manual confirmation. Bishal supplies the actual risk numbers (caps, drawdown limits); Affan/sponsor must approve the capital cap by end of W7 before this epic can go live.

Epic 5 — Observability & Reporting (W9)

Your part: a browser-based dashboard showing positions, P&L, signals, and fills, auto-refreshing under 5s, plus structured JSON logs with trace IDs. Prometheus/Grafana were explicitly deferred as "too heavy for this stage" — a lightweight custom dashboard is fine. The written findings document is mostly Khurram/Affan's job, fed by your reports.

Epic 6 — ML Serving (W9–W10, stretch)

A model-serving container on local GPU, separate from the trading loop, exposed via REST/gRPC. The ML strategy plugs into the same Strategy interface as everything else, so it automatically passes through the risk manager. Only start this if Epics 1–4 are done — it's the designated first cut if the schedule slips.

Part 8 — Working Solo: Habits That Will Save You

The BRD explicitly calls out "single-engineer dependency" as a Medium likelihood / Critical impact risk, mitigated by "keep architecture simple; document as built." A few habits make this painless rather than a burden:

  • Write decisions down as you make them. A short docs/decisions/ folder with one file per significant choice (e.g. "0001-event-bus-redis-vs-nats.md") takes two minutes and saves hours of re-explaining later.
  • Keep the README current. If a teammate (or future-you) needs to understand the system, the README + folder structure should be enough to get oriented without asking you.
  • Small, frequent commits with CI green. Even reviewing your own PRs is useful — it forces you to read your diff with fresh eyes, and CI catches mistakes before they compound.
  • A short weekly written status update (3-4 bullets: done / in progress / blocked / decisions made) posted to the Team 1 - CTB channel. This is your insurance against the bus-factor risk and keeps the PM situation from becoming a problem.
